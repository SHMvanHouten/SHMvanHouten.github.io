Override, equals and hashcode


So now we can make persons that have a first and last name and a date of birth. First we looked at how we could see if a person was already added to the list, this would require a method that see if one object equals another. Fortunately the parent class for our Person object; java.lang.Object has a method for us called equals. Unfortunately, this will not get us what we need.
When we create a new person with the Person constructor, the java engine will give that person a spot on the heap where it can access it. Now if we create a person2 with the exact same attributes it will get a different spot on the heap.
For our purposes, the first person should be exactly the same as the second person, because it is exactly the same in every attribute it has. However, the equal method does distinguish them to be different people, since they are in two different spots on the heap.
That means we will need to Override the equals method in our Person object. We could just declare the equals method in our Person class and make it do what we want because when the method is invoked it will look in the child class Person before it looks in the parent class Object. This is called overriding.
Before we do this we have to declare @Override. This serves two purposes. First, it makes it clear to other programmers (and our future selves) that we are overriding an existing method. Second, this will cause the compiler to check if we are indeed overriding a method, thus warning you if you have made a mistake and are in fact not overriding anything.

Now that we have warned the compiler that we are overriding a parent method, we can think about how we want to make the equals method do what we want it to do. If you click this link you can see what we did on line 19.
    @Override
    public boolean equals(Object obj) {
        System.out.println("Equals called to compare " + this + " with " +  obj);
        if (obj instanceof Person) {
            Person other = (Person) obj;
            if (this.firstName.equals(other.getFirstName())
                    && this.lastName.equals(other.getLastName())
                    && this.dateOfBirth.equals(other.getDateOfBirth())) {
                return true;
            }
        }
        return false;
    }
What we basically do is we compare the object we are currently in, this, to another object we give as the argument. What we compare is the identifiers for both objects, so if all the identifiers are the same, the objects will be the same. Mission accomplished!

But we’re not quite there yet. When we will be working with hash sets, which only accepts unique elements. The way it compares these elements however is it uses the hashcode for the object. Every object inherits a method that produces a hash code from java.lang.Object. The hash code will be unique to that object. This create the same problem we had with the equals method: it won’t recognize two objects that have exactly the same identifiers because they will still have different hash codes.
Solution: we must override the hashcode!
@Override
public int hashCode() {
   return this.firstName.charAt(0);
}
So now all Persons with the same starting letter in their first name will have the same hashcode (their unichar code).
Once the hash set has determined that two Person instances have the same hashcode, it will use the equals method to check if they are indeed the same, thus performing the check we need!
Whenever you change the equals method you should probably change the hashCode method too.
This all seems very complicated, but thankfully we have intelliJ to do a lot of the work for us! We can use alt insert to create an override method for equals, and because equals and hashCode go together like a horse and carriage it will suggest equals() and hashCode() in one line!

Learning Java, week 1

For a few months now I have been learning about HTML, CSS and JavaScript. While I did have the intention to keep a journal to record my experiences with JavaScript it just didn’t happen. Frankly I enjoy screwing around with the language a lot more that writing about it and after having had a few months to learn the language the backlog of things to talk about was getting a bit overwhelming.

However, now that my co-apprentice Simone and I have just switched over to Java, under the wise tutelage of Marco Beelen, I have decided to change my ways and become a good student and start to record my experiences with Java and the things I have learned about it.

On the evening of thursday the 23rd of March we had our first look at Java.
From just looking at the syntax of the language it seems pretty obvious to me that Java is an OOP (object oriented programming) language. So learning Java is also a good way to solidify my knowledge of this style of programming. Whilst I did try to do OOP in JavaScript I was still a little unsure on how to do it precisely so it will be nice to be “forced” to use it in Java.

The first terms we discussed were SDK/JDK (System/Java Development Kit) and JRE (Java Runtime Engine).
We discussed how the goal of Java’s design was for it to have as few implementation dependencies as possible. This means that you can run Java on any platforms that supports it. It does this through the JVM (Java Virtual Machine).
The process is: the code written by the programmer in .java file(s) will be compiled by the javac (java compiler) (often through the IDE) into bytecode in .class files.
These .class files all go into a .jar file.

                Java Code------>bytecode

                .Java------------->.class

                        compiler
The Java program can be run in a Java Virtual Machine. This then search for the classes in the .jar file and “compiles the bytecode into machine language so that the machine can run the code”.

Having got the introductory look at the framework out of the way we came across some other terms.
Maven, Hamcrest, Junit.
https://en.wikipedia.org/wiki/Apache_Maven

Junit is the unit testing framework we use to use test driven development.
hamcrest adds assertThat and is to our tests. https://code.google.com/archive/p/hamcrest/wikis/Tutorial.wiki

Next we had time to look at the language itself.
Although it looks quite a bit like Javascript it’s also very different and I have to say it doesn’t really feel as fluent yet as when I program with JS.
On the other hand I look forward to the fact that Java is more explicitly an object oriented language and I think learning Java will help me to further increase my knowledge of OOP as well.
In JS you just define variables with var, this could be any type of variable, so a string, a number, an object. In Java you have to define what kind of variable you are creating, so if you want to make a string you have to define it as String myString = “example”;
Also if the field or method is not inside a block scope you can also say who has access to that field or method.
public: everything has access.
protected: the class itself, classes in the package and the subclasses of the class have access.
no modifier(package lvl access): the class itself and classes in the package have access.
Private: only the class itself has access.

When creating a method you also have to specify what it should return: a string, an integer, a floating point number, etc. Void means the method doesn’t return a value.
For example:
Public String sayHello(String name){};
Private Void addScore(int amountOfScoreToAdd){};

As you can see parameters given with the method must also be specified what type they are.

That is pretty much what we discussed this lesson. We also immediately started using TDD as we were used to with JS, but that is maybe a subject for another time.
